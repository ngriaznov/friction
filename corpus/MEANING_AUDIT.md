# Meaning-preservation audit

Deterministic sample: every `corpus/manifest.jsonl` entry with `class = "llm"`,
`split = "train"`, and `id` starting with a hex character in `0`-`7`, sorted
lexicographically by `id`, first 50 taken. This range covers half the hex
alphabet (a natural, arbitrary-but-reproducible cut with no cherry-picking
knob) and yields exactly 50 of the 97 qualifying documents without needing a
separate RNG or seed. The full 97-id candidate list and the 50-id sample are
reproducible from the manifest with:

```
jq -c 'select(.class=="llm" and .split=="train")' corpus/manifest.jsonl \
  | jq -r 'select(.id | test("^[0-7]")) | .id' | sort | head -50
```

For each sampled document: ran the release `friction` CLI (`friction fix
<path> --genre <genre> --format json --suggest`) against the corpus copy,
unified-diffed input against output, and read every changed sentence for:
negation flips, dropped propositions (a deleted phrase/paragraph carrying
unique factual content), entity/number corruption, grammar breakage
(agreement, capitalization, punctuation), and markdown structure corruption
(links, code spans, list nesting). Findings that fall within this checklist
are reported as **violations** below, each tagged with the rule id that
produced them. A few additional patterns turned up that are real quality
concerns but don't fit the four checklist categories (a word-choice
imprecision, a formatting inconsistency) — those are reported separately as
**observations** and are *not* counted toward any demotion decision.

Sample size: 50/97 eligible documents (same sample as the original audit,
re-run against the working tree after the fix phase below).

## Revision note: this is a re-run after the fix phase

The original run of this audit (same 50-doc sample, same methodology) found
**5 violation instances across 3 rules**: `lexical.substitution` (3
instances: V1 `accomplish`→`do` misconjugation, V2 `showcase`→`show` entity
corruption, V3 `robust`→`solid` collocation breakage), `symmetry.
participial_closer` (1 instance: V4, promoting an imperative's closer),
and `structural.unbullet` (1 instance: V5, severe — a code-span-bearing list
merged into prose with its content dropped, contradicting the rule's own

guard so the specific violating context can no longer fire, since all four
were detectable, checkable patterns (an inherent problem, needing a

finding on `lexical.substitution`'s `specific`→`particular` entry — see
`corpus/FINGERPRINT.md`, not this audit) — and this file was regenerated by
re-running the exact same procedure against the fixed working tree. Every
section below (the walk-through of V1-V5, the per-document table, the
summary table, the human spot-check) is this re-run's own fresh output, not
hand-edited from the original.

## V1-V5, re-checked

### V1 (was) — `lexical.substitution`: grammar breakage, irregular past participle

**Doc:** `29ff22ba7de18cf6` (email, "Nonprofit Food Bank" volunteer newsletter).

> Before: `...and it's a testament to what can be accomplished when we unite for a common cause.`
> After (original run): `...It's a testament to what can be **did** when we unite for a common cause.`
> After (this run): `...it's a testament to what can be **accomplished** when we unite for a common cause.` (byte-identical to the source clause)

**Fix applied:** narrowed the guard, not demoted. `crates/friction-rules/src/families/lexical/substitution.rs`'s `is_unsafe_participle_context` skips `accomplish`'s own past-tense surface form specifically when it is immediately (or one word) preceded by a form of `"be"`/`"have"` — the passive/perfect trigger that needs the past *participle* (`"done"`), not the simple past `friction_nlp::inflect` would otherwise produce (`"did"`). A non-passive, non-perfect past-tense use of `accomplish` (no auxiliary trigger) is unaffected and still fixes normally (`"The team accomplished the goal."` → `"...did the goal."`, grammatical).

### V2 (was) — `lexical.substitution`: entity corruption, event title altered

**Doc:** `062a9d6f268e8994` (email, "Monthly Skywatcher's Newsletter").

> Before: `- **March 24th - Solar System Showcase**`
> After (original run): `- **March 24th - Solar System Show**`
> After (this run): `- **March 24th - Solar System Showcase**` (untouched)

**Fix applied:** narrowed the guard. The rule's new capitalized-span guard (`is_part_of_capitalized_span`) skips a match whenever the matched word is itself Title-Case *and* immediately adjacent to another Title-Case word — a purely orthographic, cheap-to-check signal for "probably a proper noun or title span," without needing an entity recognizer. This doc's *other*, ordinary (non-title) use of `showcase` — `"Jane's images showcase a vibrant..."` — is unaffected and still fixes to `"...images show a vibrant..."`, confirming the guard is narrow, not a blanket demotion of the `showcase → show` entry.

### V3 (was) — `lexical.substitution`: broken idiom / non-standard collocation

**Doc:** `2a2fd84658435ade` (readme, "phodedupe").

> Before: `...making it much more robust to these common image manipulations.`
> After (original run): `...making it much more **solid to** these common image manipulations.`
> After (this run): `...making it much more **robust to** these common image manipulations.` (untouched)

**Fix applied:** narrowed the guard. A new per-lemma collocation table (`COLLOCATION_GUARDS`) skips `robust` specifically when the very next word is `"to"`. Every other `robust` occurrence in-sample (`"robust error handling"`, `"robust model"`, `"robust routing"`) is unaffected and still fixes to `solid`, confirming the guard targets only the `<adj> to <noun>` frame that broke, not the entry as a whole.

### V4 (was) — `symmetry.participial_closer`: dropped instruction / wrong agency

**Doc:** `0a0197006e9ca159` (docs, "Automatic Database Backups for Corkboard").

> Before: `2. Paste the following code into the script, adjusting paths if necessary:`
> After (original run): `2. Paste the following code into the script. This adjusts paths if needed:`
> After (this run): `2. Paste the following code into the script, adjusting paths if needed:` (comma-participle structure preserved; only the unrelated `necessary → needed` lexical substitution still applies inside it)

**Fix applied:** narrowed the guard. `main_clause_is_imperative` detects a subjectless, verb-plus-object opener (a base-form or tagger-mistagged-as-bare-noun opener immediately followed by a determiner — `"Paste **the** code, ..."`) and `is_safely_promotable` now refuses to promote such a closer at all — it always falls back to `Suggest` tier (flagged, no patch), the same posture the rule already takes for a closer with no clear object. A closer whose main clause is genuinely declarative (the vast majority of firings — 7 of 8 other in-sample promotions, still promoted cleanly this run) is unaffected.

### V5 (was) — `structural.unbullet`: dropped code-span content, broken grammar, list destroyed (severe)

**Doc:** `3c9fe31d3effd9c3` (docs, "Wharfgate reverse proxy").

> Before:
> ```
> * **Service A (http://service-a.example.com):**  Handles requests starting with `/api/v1`
> * **Service B (http://service-b.example.com):** Handles requests starting with `/app`
> * **Service C (http://service-c.example.com):** Handles all other requests
> ```
> After (original run): merged into one run-on sentence, both code spans (`` `/api/v1` ``, `` `/app` ``) dropped, list structure destroyed.
> After (this run): list untouched, byte-for-byte, both code spans intact — `structural.unbullet` no longer fires on this list at all (0 firings on this doc, was 1).

**Fix applied:** narrowed the guard — and this one really was a bug, not a missing guard: the rule's own module docs already *claimed* a backtick anywhere in an item's text disqualifies the whole list, but the implementation checked that against the item's filtered *prose* text (which never includes a code span's own backtick-delimited bytes), so a code span at the very tail of an item — nothing textual after it — slipped through undetected. The fix checks the item's raw markdown source (its full block range) for a backtick, which cannot miss a code span regardless of its position within the item. `crates/friction-rules/src/families/structural/unbullet.rs`'s own regression test (`scan_does_not_match_an_item_ending_in_a_trailing_inline_code_span`) pins this down directly.

- **Negation-word count** (`not`/`never`/`no`/`n't`/`none`/`nothing`/

None of these flagged anything new. Combined with the per-document read-

methodology's checklist, applied a second time, corpus-wide rather than
only to the 5 originally-flagged documents.

## Observations (outside the 4-category checklist — not counted as violations)

Unchanged from the original run — none of these are addressed by the fix
phase above (they were explicitly out of scope: real quality concerns, not
checklist violations, and not held against any rule for a demotion
decision). Reported again here for completeness, still present in this
re-run's output:

- **`lexical.substitution`, technical-register drift: `optimize`/`optimized`
  → `improve`/`improved`.** The table entry
  (`crates/friction-rules/src/families/lexical/substitution.rs`, noted
  "optimize is a more formal synonym of improve") treats these as
  interchangeable, but in engineering prose "optimized" is often a specific
  claim (deliberately tuned for size/speed/cost) that "improved" waters
  down to a vague "made better." Still seen across the same set of sampled
  docs, e.g. doc `3c9fe31d3effd9c3`/`0859e6e20fb1cf84`: `"a fully
  functional static website optimized for deployment"` → `"...improved for
  deployment"`.
- **`lexical.substitution`, adjacent word repetition.** The substitution
  still doesn't look at nearby text, so the same replacement word can land
  twice in one sentence: doc `1ee4ee439638de43`, doc `375356550b8893e2` (see
  the original audit for the exact quotes — both still reproduce
  identically in this re-run).
- **`structural.bold_label_strip`, inconsistent application within one
  parallel list.** Unaffected by this fix phase; doc `1ee4ee439638de43` and
  doc `3154ba801fd8427c` still show the same inconsistent unbolding pattern
  the original audit documented.
- **`rhythm.split`, same inconsistency pattern.** Doc `447496100b5f2ff8`
  still has the same partially-applied semicolon-to-period pattern the
  original audit documented.
