# Catastrophic Backtracking in Regex Patterns with Nested Quantifiers

## Problematic Pattern

The following regex pattern matches repeated groups of optional characters:

```regex
(?:(?:a|b)*c)*(?:d|e)*
```

This pattern appears to match strings that contain one or more occurrences of the character 'c', followed by zero or more occurrences of either the character 'd' or 'e'.

## Why it Causes Catastrophic Backtracking

The issue with this pattern lies in its use of nested quantifiers. The outermost group `*` is applied to the entire match, which means that the regex engine must try all possible combinations of characters from the beginning of the string until a match is found. This leads to an exponential number of possibilities being explored.

Inside the outermost group, there are two more groups with quantifiers: `*(?:a|b)*c` and `(?:d|e)*`. The first group uses a `*` quantifier followed by an optional character class (`(?:a|b)`), which allows for zero or one occurrence of either 'a' or 'b'. This group is then applied to the character 'c', effectively allowing for zero occurrences of 'c'.

The second group, `(?:d|e)*`, applies another `*` quantifier followed by an optional character class (`(?:d|e)`). This allows for zero or one occurrence of either 'd' or 'e'. When combined with the first group, this creates a situation where the regex engine must try all possible combinations of characters from the beginning of the string.

## Exponential Blowup

The problem arises when the input string is long enough that there are more possibilities than can be handled by the regex engine in a reasonable amount of time. The number of possible combinations grows exponentially with the length of the input string, leading to catastrophic backtracking and, ultimately, a hang.

For example, consider an input string of 1000 characters. If each character has two possible choices (either 'a' or 'b', or either 'd' or 'e'), there are $2^{1000}$ possible combinations. This is far beyond the capabilities of most regex engines to process in a reasonable amount of time.

## Rewritten Pattern

To avoid catastrophic backtracking, we can rewrite the pattern using possessive quantifiers (`+` instead of `*`). The rewritten pattern is as follows:

```regex
(?:(?:a|b)+c)?(?:d|e)?
```

In this version, the first group uses a possessive quantifier `+(?:a|b)` to match one or more occurrences of either 'a' or 'b'. This prevents backtracking by ensuring that once a match is found for one character, the regex engine does not try other characters again.

The second group remains unchanged with `(?:d|e)?`, which allows for zero or one occurrence of either 'd' or 'e'.

By using possessive quantifiers, we avoid the exponential blowup caused by nested quantifiers and ensure that the regex engine can process long input strings efficiently.
