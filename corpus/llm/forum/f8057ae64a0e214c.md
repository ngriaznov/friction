### How to Write a Systemd Unit File for a Custom Long-Running Daemon

When managing long-running services in Linux using `systemd`, it's crucial to configure the service properly so that it starts, runs efficiently, and handles failures gracefully. This answer will walk you through creating a systemd unit file for your custom daemon, focusing on key configurations such as `Restart=on-failure`, `WatchdogSec` with `sd_notify` keepalives, and understanding the differences between `Type=simple` and `Type=notify`.

#### 1. Basic Structure of a Systemd Unit File

A basic systemd service unit file has a simple structure that includes various directives to control its behavior:

```ini
[Unit]
Description=Custom Daemon Service
After=network.target

[Service]
User=myuser
ExecStart=/path/to/your-daemon
Restart=on-failure
Type=simple
WorkingDirectory=/var/lib/mydaemon
```

#### 2. Restart Configuration (`Restart`)

The `Restart` directive determines how the service behaves when it fails to start or exits with a non-zero status:

- `no`: Do not restart on failure (default).
- `on-success`: Only restart if the previous exit status was 0.
- `on-failure`: Restart the service after any failure.

For a long-running daemon, setting `Restart=on-failure` ensures that if your application crashes, it will be restarted automatically, improving resilience:

```ini
[Service]
...
Restart=on-failure
```

#### 3. Watchdog Mechanism (`WatchdogSec`)

The `WatchdogSec` directive allows you to configure a watchdog timer for the service. If the daemon does not send any notification during this time period, it is considered dead and will be restarted.

To use `sd_notify`, which is part of the `systemd-daemon`, your application needs to periodically send "keepalive" messages back to systemd using `sd-daemon Notify` or similar mechanisms. This can be done in various languages; for example, in Python:

```python
import os

def keep_alive():
    os.system('echo "STATUS=Running" | sd-daemon --listen')
```

Here’s how you might configure the unit file to use `WatchdogSec` and ensure your application sends keepalives:

```ini
[Service]
...
Restart=on-failure
WatchdogSec=10s  # Send a keepalive every 10 seconds.
TimeoutStopSec=30s  # Wait for the process to stop gracefully before killing it.
```

#### 4. Service Type (`Type`)

The `Type` directive specifies the type of service unit, which can affect how `systemd` manages the service:

- `simple`: The default option, where the service runs in its own process and exits on completion or when signaled to stop.
- `notify`: Similar to `simple`, but sends a `STATUS=RUNNING` message to systemd using `sd_notify`.

For applications that need more control over their lifecycle and status reporting, you might choose `Type=notify`. This is useful for applications that want to report their state changes:

```ini
[Service]
...
Restart=on-failure
WatchdogSec=10s
TimeoutStopSec=30s
Type=notify  # Notify systemd of the service's state.
```

#### Example Unit File

Combining these elements, here is an example unit file for a custom daemon:

```ini
[Unit]
Description=Custom Daemon Service
After=network.target

[Service]
User=myuser
ExecStart=/path/to/your-daemon
Restart=on-failure
WatchdogSec=10s
TimeoutStopSec=30s
Type=simple  # or notify if your application supports it.
WorkingDirectory=/var/lib/mydaemon

# Keepalive mechanism, adjust as necessary for your application
[Install]
WantedBy=default.target
```

This setup ensures that the daemon is managed effectively by `systemd`, with automatic restarts on failure and periodic notifications to keep the watchdog alive. Adjustments may be needed based on specific requirements of your custom daemon.

### Conclusion

Properly configuring a systemd unit file for long-running daemons involves setting appropriate restart policies, enabling watchdog mechanisms, and choosing the correct service type. These configurations help ensure that your application runs reliably and can recover from failures gracefully.
