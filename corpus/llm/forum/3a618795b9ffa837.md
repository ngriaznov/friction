**When does a covering index avoid a table lookup in MySQL?**

A covering index is an index on a table that contains all the columns needed to answer a query, thereby avoiding the need for a table lookup. In other words, if you have an index that includes all the columns used in the WHERE and SELECT clauses of your query, the database can retrieve the required data directly from the index without accessing the underlying table.

Let's walk through a sample scenario to illustrate this concept.

**Initial Query**

Suppose we have a table called `employees` with the following schema:
```sql
CREATE TABLE employees (
  id INT PRIMARY KEY,
  name VARCHAR(255),
  department_id INT,
  salary DECIMAL(10,2)
);
```
We want to retrieve the names and salaries of all employees in the sales department. Our initial query is:
```sql
SELECT name, salary FROM employees WHERE department_id = 1;
```
Let's examine the EXPLAIN output for this query:
```sql
EXPLAIN SELECT name, salary FROM employees WHERE department_id = 1\G
*************************** 1. row ***************************
id: 1
select_type: SIMPLE
table: employees
type: ALL
possible_keys: NULL
key: NULL
key_len: NULL
ref: NULL
rows: 1000
filtered: 100.00
Extra: Using where; Using filesort
```
In the EXPLAIN output, we see that MySQL is using the `WHERE` clause to filter the data and performing a file sort to retrieve the required rows.

**Adding a Composite Index**

To avoid the table lookup, let's create a composite index on the `department_id` column and add it to our index:
```sql
CREATE INDEX idx_department_id ON employees (department_id, name, salary);
```
Now, when we re-execute our original query with the new index in place, the EXPLAIN output changes significantly:
```sql
EXPLAIN SELECT name, salary FROM employees WHERE department_id = 1\G
*************************** 1. row ***************************
id: 1
select_type: SIMPLE
table: employees
type: index_merge
possible_keys: idx_department_id
key: idx_department_id
key_len: 4
ref: NULL
rows: 10
filtered: 100.00
Extra: Using index; Using where
```
In this revised EXPLAIN output, we notice the following changes:

* The `type` field now shows `index_merge`, indicating that MySQL is using a merge of indexes to retrieve the data.
* The `key` field lists our composite index `idx_department_id`.
* The `Extra` column no longer mentions "filesort," as the index provides all necessary columns for the query.

**Why the change?**

With the new composite index in place, MySQL can now use the index to satisfy the query directly. Since the index contains both the `department_id` and `name` columns used in the WHERE clause, the database doesn't need to access the underlying table to retrieve the data.

The absence of "Using filesort" in the revised EXPLAIN output demonstrates that the index is now providing all necessary information for the query, eliminating the need for additional processing.

**Conclusion**

In conclusion, a covering index can significantly improve query performance by avoiding table lookups. By creating an index on columns used in both the WHERE and SELECT clauses of your query, you can enable MySQL to retrieve data directly from the index, reducing the load on disk I/O and CPU resources.

Remember that adding composite indexes requires careful consideration of the specific queries and use cases for your application. Always analyze your EXPLAIN output and understand how different indexing strategies impact performance.
