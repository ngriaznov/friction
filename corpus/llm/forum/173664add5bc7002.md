# Covering Indexes in MySQL: Avoiding Table Lookups

In MySQL, an index can be used to improve the performance of a query by reducing the number of disk accesses required to retrieve data from a table. One type of index that is particularly useful in this regard is the covering index.

## What is a Covering Index?

A covering index is an index on one or more columns of a table where all the rows needed for a given query are contained within the indexed column(s) themselves. In other words, if you can retrieve all the necessary data from the index without having to access the underlying table, then the index is said to be "covering" the data.

## When Does a Covering Index Avoid Table Lookups?

A covering index will avoid table lookups in the following situations:

1.  **When the query only selects columns that are covered by the index**: If you're selecting only those columns that are already included in the index, MySQL can use the index to retrieve all the necessary data without having to access the underlying table.
2.  **When the query uses a SELECT statement with a WHERE clause that is covered by the index**: If your query includes a WHERE clause that matches one of the indexed columns, and you're selecting only those columns that are included in the index, MySQL can use the index to retrieve all the necessary data.

## Sample EXPLAIN Output Before Adding Composite Index

Let's consider an example to illustrate this concept. Suppose we have the following table:

```sql
CREATE TABLE customers (
    id INT,
    name VARCHAR(255),
    email VARCHAR(255)
);
```

And let's insert some sample data:

```sql
INSERT INTO customers (id, name, email) VALUES
(1, 'John Doe', 'john.doe@example.com'),
(2, 'Jane Doe', 'jane.doe@example.com'),
(3, 'Bob Smith', 'bob.smith@example.com');
```

Now, let's run a query that selects the `name` and `email` columns for all customers where `id = 1`:

```sql
SELECT name, email FROM customers WHERE id = 1;
```

Running this query without any indexes will produce an EXPLAIN output like this:

```sql
+----+--------+--------+--------+---------------+---------+-----------+--------+--------+
| id | select_type | table   | type    | possible_keys | key     | key_len   | ref   | rows  |
+----+--------+--------+--------+---------------+---------+-----------+--------+--------+
| 1  | SIMPLE    | customers| ALL     | NULL          | NULL    | NULL      | NULL  | 3     |
| 2  | DERIVED   | NULL    | NULL    | NULL          | NULL    | NULL      | NULL  | 0     |
+----+--------+--------+--------+---------------+---------+-----------+--------+--------+
```

In this output, the `key` column is `NULL`, indicating that MySQL cannot find a matching index on the table. This means that MySQL will have to access the underlying table (`customers`) to retrieve all the necessary data.

## Sample EXPLAIN Output After Adding Composite Index

Now, let's create a composite index on the `id` and `name` columns:

```sql
CREATE INDEX idx_name_id ON customers (id, name);
```

With this index in place, we can run the same query again, but this time with an EXPLAIN output like this:

```sql
+----+--------+--------+--------+---------------+---------+-----------+--------+--------+
| id | select_type | table   | type    | possible_keys | key     | key_len   | ref   | rows  |
+----+--------+--------+--------+---------------+---------+-----------+--------+--------+
| 1  | SIMPLE    | customers| range   | NULL          | idx_name| 10        | const| 1     |
| 2  | DERIVED   | NULL    | NULL    | NULL          | NULL    | NULL      | NULL  | 0     |
+----+--------+--------+--------+---------------+---------+-----------+--------+--------+
```

In this output, the `key` column is now `idx_name`, indicating that MySQL has found a matching index on the table. This means that MySQL can use the index to retrieve all the necessary data without having to access the underlying table.

## Using Index vs Using where; Using filesort Distinction

The key difference between these two EXPLAIN outputs lies in how MySQL decides which index to use and how much data it needs to scan.

*   In the first output, MySQL is unable to find a matching index on the `id` column alone, so it has to access the underlying table. This indicates that MySQL cannot determine whether the query will benefit from using an index.
*   In the second output, MySQL is able to use the composite index because it can determine that the query will benefit from using the index. However, since the `key_len` column is 10 (which represents the length of the index), we know that MySQL has to scan less data.

The `Extra` column also provides some additional information about how MySQL decides which index to use and what operations it needs to perform:

*   In the first output, the `Extra` column is `Using filesort`, indicating that MySQL needs to sort the results.
*   In the second output, the `Extra` column is `Using where; Using filesort`, indicating that MySQL can use an index but still needs to sort the results.

In conclusion, a covering index in MySQL can significantly improve the performance of queries by avoiding table lookups. By creating indexes on columns used in WHERE clauses and SELECT statements, you can reduce the amount of data MySQL needs to scan and improve overall query performance.

**When to create Covering Indexes:**

*   When selecting only columns that are included in the index.
*   When using a WHERE clause that matches one of the indexed columns.
*   To eliminate table lookups for certain queries.

By creating covering indexes, you can significantly improve the performance of your MySQL queries and reduce the amount of data that needs to be scanned.
