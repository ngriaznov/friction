Writing a systemd Unit File for a Long-Running Daemon
=====================================================

Systemd is a powerful tool for managing system services, including long-running daemons like yours. Writing an effective systemd unit file requires understanding how to configure the daemon's behavior, handle restarts, and implement watchdogs. In this answer, we'll cover the basics of writing a systemd unit file for your custom long-running daemon.

### Restart Policy: on-failure

When writing a systemd unit file, you can specify how the system should behave when the service fails or restarts. The `Restart` policy controls whether the service is restarted immediately after it fails or waits until the next boot cycle to attempt another restart.

To use the `on-failure` policy, simply add the following line to your unit file:

```bash
Restart=on-failure
```

This tells systemd to restart the service if it fails, but only after waiting for a short period of time. You can adjust this timeout by using the `RestartSec` option.

### Watchdog Configuration

A watchdog is a mechanism that periodically checks on a service and restarts it if it becomes unresponsive. In systemd, you can configure a watchdog to check on your daemon every second (the default interval) or at a custom interval specified with the `WatchdogSec` option.

To implement a watchdog using `sd_notify`, you'll need to modify your daemon's code to send notifications in response to certain events. Here are the steps:

1.  Create an `notify` method in your daemon that sends an `SDNotify` notification when it becomes ready or needs attention.
2.  In your systemd unit file, add the following line:

    ```bash
WatchdogSec=10s
```

    This specifies that the watchdog should check on the service every 10 seconds.
3.  Use the `sd_notify` command to send notifications from within your daemon. When your daemon is ready or needs attention, use `sd_notify --notify READY` or `sd_notify --notify NEED ATTENTION` to signal to systemd.

Here's an example of how you might modify your daemon code:

```c
#include <stdio.h>
#include <stdlib.h>

int main() {
    // Your daemon's initialization and main loop here...

    while (1) {
        if (some_condition_is_met()) {
            sd_notify(SD_NOTIFY_READY);
        } else {
            sd_notify(SD_NOTIFY_NEED_ATTENTION);
        }

        // Do some work...
    }
}
```

In this example, the `sd_notify` command is used to send notifications when your daemon becomes ready or needs attention.

### Type Options

When writing a systemd unit file, you can specify how the service should be managed with the `Type` option. The most common types are:

*   **simple**: This type tells systemd to start and stop the service as needed.
*   **notify**: This type is similar to `simple`, but it allows you to notify other services when your daemon starts or stops.

Here's an example of how to use each type in a unit file:

```bash
# Simple Type
[Unit]
Description="My Long-Running Daemon"
After=network.target

[Service]
Type=simple
ExecStart=/path/to/daemon
Restart=on-failure

# Notify Type
[Unit]
Description="My Long-Running Daemon (Notify)"
After=network.target

[Service]
Type=notify
ExecStart=/path/to/daemon
Restart=on-failure
```

In the `simple` type example, systemd will start and stop your daemon as needed. In the `notify` type example, systemd will only start your daemon when it's ready to start, and notify other services that your daemon is up or down.

### Conclusion

Writing an effective systemd unit file for a long-running daemon requires understanding how to configure the daemon's behavior, handle restarts, and implement watchdogs. By using the `on-failure` policy, configuring a watchdog with `sd_notify`, and choosing the right `Type` option, you can create a robust service management system that meets your daemon's needs.

Here is an example of a complete unit file for your long-running daemon:

```bash
[Unit]
Description="My Long-Running Daemon"
After=network.target

[Service]
Type=simple
Restart=on-failure
ExecStart=/path/to/daemon
WatchdogSec=10s

[Install]
WantedBy=multi-user.target
```

This example uses the `simple` type, restarts on failure with a 10-second watchdog interval, and starts when the system boots into the multi-user target.
