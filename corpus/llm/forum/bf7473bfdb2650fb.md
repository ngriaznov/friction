## How to Write a Systemd Unit File for a Long-Running Daemon

This answer explains how to write a systemd unit file for your custom long-running daemon, covering essential concepts like `Restart=on-failure`, `WatchdogSec` with `sd_notify` keepalives, and the difference between `Type=simple` and `Type=notify`.

**Understanding Systemd Unit Files**

Systemd uses unit files to define services, targets, timers, and other system components. A unit file is a text file containing directives that specify how a service should behave. 

For long-running daemons, the most common type of unit file is a `*.service` file. It defines the daemon's name, executable path, working directory, environment variables, startup and shutdown scripts, and other relevant parameters.

**Restart Policy: `Restart=on-failure`**

The `Restart=` directive controls how systemd handles service failures. 

* **`Restart=always`**: The service will be restarted unconditionally after every failure.
* **`Restart=on-failure`**: The service will only be restarted if it exits with a non-zero exit code (indicating an error). This is the recommended setting for most daemons.

**Watchdog Timeout: `WatchdogSec` and `sd_notify` Keepalives**

Systemd uses a watchdog mechanism to monitor running services. If a service doesn't send a "keepalive" signal within the specified `WatchdogSec` interval, systemd assumes it has crashed and restarts it.

For long-running daemons, it's crucial to implement keepalives. The `sd_notify` API provides a convenient way to do this:

* Your daemon should call `sd_notify(0, "KEEPALIVE")` periodically (e.g., every few seconds) to signal systemd that it's still running.
* This prevents unnecessary restarts due to watchdog timeouts.

**Service Types: `Type=simple` vs. `Type=notify`**

Systemd offers two main service types:

* **`Type=simple`**: The simplest type, suitable for daemons that execute a single command and exit. It doesn't support keepalives or other advanced features.
* **`Type=notify`**: More versatile, allowing for complex startup and shutdown scripts, `sd_notify` integration, and finer control over service behavior.

For long-running daemons, **`Type=notify`** is generally preferred due to its flexibility and support for keepalives.

**Example Unit File (`mydaemon.service`)**

```ini
[Unit]
Description=My custom daemon
After=network.target

[Service]
Type=notify
User=youruser
WorkingDirectory=/path/to/daemon
ExecStart=/path/to/mydaemon
Restart=on-failure
WatchdogSec=30s
StandardOutput=journal

[Install]
WantedBy=multi-user.target
```

**Explanation:**

* **`Description`**: A brief description of the daemon.
* **`After=network.target`**: Ensures the network is up before starting the daemon.
* **`Type=notify`**: Specifies the service type.
* **`User`**: Sets the user account under which the daemon runs.
* **`WorkingDirectory`**: Defines the daemon's working directory.
* **`ExecStart`**: The path to the daemon executable.
* **`Restart=on-failure`**: Restarts the daemon on failure.
* **`WatchdogSec`**: Sets the watchdog timeout to 30 seconds.
* **`StandardOutput=journal`**: Logs output to systemd journal.

**Remember:** Replace placeholders with your actual values and ensure your daemon implements `sd_notify` keepalives for reliable operation.
