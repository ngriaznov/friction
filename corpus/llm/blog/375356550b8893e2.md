**Switching Gear: Why Our Five-Person Engineering Team Migrated from Gerrit to GitHub Pull Requests**

In the ever-evolving landscape of software development, tools and processes are continually evolving too. For our five-person engineering team, we recently made a significant shift in how we handle code reviews—from using the long-standing Gerrit platform to adopting GitHub pull requests. This transition wasn't just about switching tools; it was about enhancing our workflow, making collaboration smoother, and ensuring that everyone on the team could contribute effectively.

### The Journey with Gerrit

Gerrit is a robust tool for managing code reviews in large-scale projects. It has served us well over the years, particularly because of its powerful branching model and integration capabilities. However, as we grew more accustomed to Git workflows and the increasing popularity of GitHub among our team members, we began to recognize some limitations that were no longer working in our favor.

#### What We Missed About Gerrit

1. **Learning Curve for New Team Members:**
   While Gerrit is powerful, it has a steeper learning curve compared to GitHub. Our team includes several new members who joined recently, and they found the initial setup and navigation through Gerrit more challenging than expected.

2. **Integration with Modern Tools:**
   Over time, our workflow evolved, incorporating various tools like Codecov for code coverage and GitHub Actions for CI/CD pipelines. While Gerrit supported these integrations to some extent, it wasn’t as seamless or intuitive compared to GitHub’s ecosystem.

3. **Simplicity in Pull Requests:**
   Pull requests on GitHub offer a more streamlined approach to reviewing changes. They are easier to understand and manage, especially when dealing with multiple branches and pull request reviews. The visual feedback provided by GitHub makes it simpler for everyone to see the impact of proposed changes.

4. **Community Support and Updates:**
   GitHub is widely used across the industry, which means that there's a larger community of users who contribute ideas, plugins, and solutions. This support network can be invaluable when facing issues or looking for new features.

### Making the Switch

After carefully evaluating both tools, we decided to migrate our code review process from Gerrit to GitHub pull requests. Here’s why this switch made sense for us:

1. **Reduced Friction:**
   Transitioning from Gerrit to GitHub meant that onboarding new team members became less daunting. They could quickly pick up the workflow and start contributing without feeling overwhelmed by the initial setup.

2. **Enhanced Collaboration:**
   The pull request feature in GitHub allowed for better collaboration among our small but dynamic team. It was easier to engage multiple reviewers and get feedback, which led to higher-quality code submissions faster.

3. **Streamlined Workflow:**
   The pull request model on GitHub simplified the review process. We could easily link related issues, track comments, and merge changes into the main branch—all within a familiar Git workflow.

4. **Built-in CI/CD Integration:**
   One of the key benefits we noticed was the seamless integration with our existing CI/CD setup using GitHub Actions. This allowed us to automatically run tests and check code coverage directly in the pull request, ensuring that every change met our quality standards before it was merged.

5. **Improved Communication:**
   The comments and discussions within GitHub pull requests facilitated better communication among team members. We could discuss changes, clarify doubts, and make adjustments more efficiently compared to the more rigid Gerrit workflow.

### Conclusion

Migrating from Gerrit to GitHub pull requests wasn’t just about changing tools; it was about optimizing our development process for a smaller, yet highly collaborative team. While we miss some of the advanced features in Gerrit, the benefits gained in simplicity and efficiency far outweighed those limitations. If your team is considering a similar transition, or even if you're looking to streamline your code review processes within an existing framework, exploring GitHub pull requests might just be the way forward.

By making this switch, we not only improved our development process but also enhanced the overall experience for everyone on the team. It's a testament to how staying agile and open to change can lead to more productive and satisfying software projects.
