## My Tiny Kubernetes Playground: Running K8s on Three Raspberry Pi 4s

For years, I’ve been fascinated by Kubernetes – the orchestrator that lets you manage containerized applications at scale. The thought of deploying complex systems with a simple command always seemed like a beautiful abstraction. But it felt… distant. Expensive. A world away from my tinkering habits. Then, a conversation with a friend sparked an idea: why not build my own small, manageable Kubernetes cluster right in my home?  

The result? A surprisingly rewarding (and occasionally frustrating) journey into the heart of container orchestration using just three Raspberry Pi 4 boards. It’s far from production-ready, but it's become an incredible learning lab and a fantastic way to demystify K8s without breaking the bank. Let me walk you through how I built this little beast, highlighting the challenges – particularly around networking and storage – and what I actually managed to get running on it.

**The Hardware Setup: Three Pi 4s and a Lot of Patience**

I opted for three Raspberry Pi 4 models with 4GB of RAM each. The price was right, and the processing power is surprisingly decent for this scale. Setting up the initial network was relatively straightforward – all connected to my home Wi-Fi (a mesh network helps with stability).  I used Raspberry Pi OS Lite (64-bit) on each device, minimizing overhead and simplifying things. 

The core of the cluster consists of:

* **Master Node (Pi 1):** This handles the control plane – the Kubernetes API server, scheduler, etc.
* **Worker Nodes (Pi 2 & Pi 3):** These run the applications you deploy.  I designated Pi 2 and Pi 3 as worker nodes.


**Networking: A Surprisingly Complex Puzzle**

This is where things got tricky. Initially, I tried using Kubernetes' built-in networking solution, Calico. It’s powerful but complex to configure, especially for a small cluster. The documentation wasn't always clear on the best practices for Pi networks, and I spent *hours* troubleshooting DNS resolution issues – seemingly random failures that were incredibly difficult to diagnose. 

Ultimately, I realized the network itself was the bottleneck. Wi-Fi isn’t ideal for inter-node communication, leading to latency spikes and occasional connection drops.  I decided to take a more direct approach using Flannel, a simple overlay network solution designed for Kubernetes. Flannel uses VXLAN (Virtual Extensible LAN) to create a virtual network on top of my existing Wi-Fi network. 

This drastically improved connectivity between the nodes, though I still noticed occasional slowdowns under heavy load.  I experimented with different Wi-Fi channels and ensured strong signal strength across all devices – essential for reliable Kubernetes operation.



**Storage: The Biggest Headache (and Why It Matters)**

The biggest challenge wasn't the networking; it was storage. Kubernetes relies heavily on persistent volumes to store application data. This is where things got seriously complicated. 

My initial idea was simple: mount a shared directory across all three Pi’s using NFS (Network File System).  This quickly devolved into a chaotic mess of permissions errors, file locking issues, and generally unreliable storage. Kubernetes kept failing to create persistent volumes because the nodes couldn't consistently access the same files. 

I realized I was forcing a centralized approach onto a distributed system. The inherent limitations of three small Pi’s with Wi-Fi networks simply weren't conducive to reliable shared storage.  

After much experimentation, I settled on a hybrid solution: 
* **Local Storage:** Each node has a local SSD connected via USB. This is where I store the data for my applications. It provides fast access and eliminates the network dependency.
* **Network File System (NFS) with careful configuration:**  I configured an NFS server on one of the Pi’s, sharing a directory across all nodes. However, this was primarily used for storing Kubernetes configurations – manifests, Helm charts, etc., rather than application data itself. 

It wasn't perfect, but it provided a workable solution that addressed the core storage requirements of my cluster.



**What I Actually Ran: A Bit of Fun and a Lot of Learning**

Despite the challenges, I was able to deploy several applications onto this tiny Kubernetes cluster. Here’s what I ended up running:

* **Traefik Ingress Controller:**  I used Traefik to expose my applications externally, handling incoming requests and routing them to the appropriate services within the cluster.
* **Nginx Web Server (as a Service):**  I deployed an Nginx web server as a Kubernetes service, allowing me to experiment with reverse proxy configurations and load balancing.
* **Simple Python Application:** I created a basic Flask application – essentially a "Hello World" example – to demonstrate deploying and scaling applications within the cluster. 
* **A Minecraft Server (Experiment):**  This was purely for fun! I managed to get a small instance of Minecraft running, demonstrating the ability to scale the server based on player load.



**Lessons Learned and Future Plans**

Building this home Kubernetes cluster has been an incredibly valuable learning experience. Here's what I took away:

* **Start Small:** Don’t try to replicate complex production environments initially.  This tiny cluster is a perfect starting point for understanding the fundamentals of K8s.
* **Networking Matters:** Invest time in getting your network right – it's crucial for Kubernetes performance and stability. Flannel was a much better choice than Calico for my setup.
* **Storage Complexity:** Don’t underestimate the challenges of storage in a distributed environment, especially with limited resources. Local SSDs are often a more reliable solution than shared network storage.
* **Kubernetes is Powerful (But Requires Effort):**  It's not magic; it takes time and effort to configure and manage.



Moving forward, I’m planning on:

* **Adding More Nodes:** Expanding the cluster with additional Raspberry Pi 4 boards would allow for more complex deployments and better scalability.
* **Exploring Storage Solutions:** Investigating persistent volume solutions like Rook or Portworx could provide a more robust and efficient storage layer.
* **Deploying More Complex Applications:** I’m interested in experimenting with deploying more sophisticated applications, such as databases or message queues.

Ultimately, my tiny Kubernetes playground isn't about building a production-ready system; it's about gaining practical experience, deepening my understanding of container orchestration, and enjoying the process of tinkering with this fascinating technology.  If you’ve ever been intimidated by Kubernetes, I highly recommend giving this a try – you might be surprised at how much you can learn!
