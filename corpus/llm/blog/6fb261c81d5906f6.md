##  From Gerrit's Rigor to GitHub's Flow: A Five-Person Team's Code Review Journey

For years, our five-person engineering team relied on Gerrit for code reviews. We appreciated its robust features, structured workflow, and commitment to clean, maintainable code. But as our projects evolved and we sought greater agility, we realized that Gerrit’s strengths had become somewhat of a burden. 

So, after careful consideration, we switched to GitHub pull requests (PRs). This wasn't an easy decision, but one we believe ultimately benefits our team and our workflow. Let's delve into the reasons behind this shift and explore both what we missed about Gerrit and how things have become easier with GitHub PRs.

**Gerrit: The Fortress of Code Quality**

We won’t sugarcoat it - Gerrit is a powerful tool. Its strict change submission process, detailed review comments, and granular access control helped us maintain incredibly high code quality. We loved the ability to easily track changes, enforce coding standards, and ensure every piece of code underwent thorough scrutiny before merging. 

Gerrit's "no merge without a review" policy forced us to engage in thoughtful discussions and learn from each other. This led to cleaner code, fewer bugs, and a shared understanding of project architecture. The dedicated review interface fostered focused feedback and minimized distractions from the main development workflow.

**The Drawbacks of Gerrit:**

While we cherished its strengths, Gerrit's rigidity started to feel like an obstacle as our projects became more dynamic. 

* **Steep learning curve:**  Gerrit requires a significant time investment to learn and master, especially for new team members.
* **Limited integration:** Its lack of seamless integration with popular tools like CI/CD pipelines and communication platforms created workflow inefficiencies.
* **Clunky user experience:** While functional, Gerrit's interface wasn't always intuitive or user-friendly.

**GitHub PRs: Embracing Flow and Collaboration**

Switching to GitHub PRs was a deliberate move towards greater agility, transparency, and ease of use. Here's what we gained:

* **Streamlined workflow:** The integration with our existing CI/CD pipelines and communication channels smoothed out our development process. 
* **Enhanced collaboration:**  GitHub PRs foster open discussions and encourage active participation from everyone on the team.
* **Simpler user experience:** The intuitive interface and familiar layout made it easy for new team members to jump in and contribute.

**What We Miss (and How We Adapt):**

While GitHub PRs have significantly improved our workflow, we do miss some aspects of Gerrit:

* **Granular access control:**  Gerrit's fine-grained permissions allowed us to define precise roles and responsibilities. While GitHub offers similar features, they are less granular. We've adapted by implementing more robust project documentation and communication strategies.
* **Dedicated review interface:** Gerrit's focused review interface minimized distractions. With GitHub PRs, we need to be more conscious of managing our notifications and staying on top of discussions within the broader platform.

**Looking Forward: A Continuous Journey**

Our switch from Gerrit to GitHub PRs represents a continuous effort to optimize our workflow and adapt to evolving needs. We're constantly exploring new tools and techniques to improve collaboration, code quality, and project efficiency. This journey is an ongoing exploration, and we're excited to see what the future holds!
