Title: From Gerrit to GitHub Pull Requests: Our Journey with Code Review Tools

As an engineering team of five, we've been fortunate enough to have had the opportunity to work with various code review tools throughout our development journey. One tool that has stood out in particular is Gerrit, which we used for over three years. However, after careful consideration and experimentation, we made the decision to switch from Gerrit to GitHub pull requests. In this blog post, we'll delve into what we missed about Gerrit and what got easier when we made the switch.

What We Missed About Gerrit

Before making the switch, we evaluated our current workflow and identified areas where Gerrit fell short. One of the primary reasons we wanted to move away from Gerrit was its clunky interface. The web interface, although functional, lacked the modern feel that many of us are accustomed to in today's development environment. Furthermore, Gerrit's user interface could become overwhelming with multiple branches and changes, making it difficult for team members to navigate.

Another area where we struggled with Gerrit was the lack of integration with our CI/CD pipeline. Our Jenkins setup required manual intervention to push code from Gerrit into the pipeline, which introduced a layer of complexity that we wanted to avoid. With GitHub pull requests, integrating with Jenkins became seamless, as the two tools are now tightly coupled.

Finally, Gerrit's review workflow could become cumbersome, especially when dealing with large projects or multiple reviewers. The tool relied heavily on manual configuration and setup, which made it challenging for new team members to onboard. In contrast, GitHub pull requests offer a more streamlined process for setting up reviews, making it easier for new team members to contribute.

What Got Easier

So, what did we gain when we switched from Gerrit to GitHub pull requests? One of the most significant benefits was the improved user interface. The modern design and intuitive navigation made it easy for our team to manage code changes and collaborate on reviews.

Another advantage we discovered was the ease of integration with our CI/CD pipeline. With GitHub's built-in support for integrating with Jenkins, our workflow became more streamlined, reducing manual intervention and errors.

Perhaps most notably, however, was the simplified review process. The automated workflows in GitHub pull requests eliminated many of the manual configurations required in Gerrit, making it easier to onboard new team members and manage code changes.

Additionally, we found that GitHub's pull request feature allowed us to take advantage of some advanced features like code reviews with multiple reviewers, as well as a built-in diff viewer for reviewing changes. These features greatly enhanced our collaboration process and ensured that every team member had access to the necessary tools to contribute effectively.

Best Practices for Making the Switch

If you're considering making the switch from Gerrit to GitHub pull requests, here are some best practices we recommend:

1.  **Plan Ahead**: Before making the switch, plan out your review workflow. Identify what features you'll need and how they'll fit into your current process.
2.  **Test Thoroughly**: Make sure to test thoroughly with a small group of team members before making the switch.
3.  **Document Process**: Document your new review workflow so that all team members are on the same page.

In conclusion, our experience switching from Gerrit to GitHub pull requests has been overwhelmingly positive. By identifying what we missed about Gerrit and understanding how GitHub's tool would simplify our code review process, we were able to make a seamless transition that improved collaboration and efficiency within our engineering team.
