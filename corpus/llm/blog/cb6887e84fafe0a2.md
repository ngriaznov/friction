# Building a Small Home Kubernetes Cluster: Lessons Learned

In this blog post, I'll share my journey in setting up a small Kubernetes cluster using three Raspberry Pi 4 boards at home. This project not only provides an opportunity to learn about container orchestration but also serves as a fun and practical way to explore the power of Kubernetes on a shoestring budget. From networking setup to storage challenges, here are some key insights and lessons I learned along the way.

## The Setup

My goal was simple: to create a small, self-contained Kubernetes cluster that could run basic applications like a web server or a database instance. Given the constraints (limited resources), I decided to use three Raspberry Pi 4 boards, each with 2GB of RAM and a single-core processor running at 1.5 GHz.

### Hardware Requirements

- **3 x Raspberry Pi 4** - Each board came equipped with 2GB of RAM.
- **Power Supply** - A reliable power source for the Pis.
- **Storage** - SD cards to install operating systems and Kubernetes components.
- **Networking** - Ethernet cables for stable connections.

## Networking Setup

Setting up the network was one of the more straightforward parts, but it required careful planning. I opted for a simple setup using static IP addresses on each Pi to ensure consistent communication within the cluster.

### Step 1: Assign Static IPs

I set up each Raspberry Pi with a static IP address in the same subnet (e.g., `192.168.0.5`, `192.168.0.6`, and `192.168.0.7`). This made it easy to identify them within my network.

### Step 2: Configure Networking

Each Pi was connected via Ethernet to a common switch, which in turn was connected to the main router. I used `netplan` on each Raspberry Pi to define the static IP addresses:

```yaml
# /etc/netplan/01-netcfg.yaml
network:
  version: 2
  ethernets:
    eth0:
      dhcp4: no
      addresses:
        - 192.168.0.XXX/24
      gateway4: 192.168.0.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4
```

Replace `XXX` with the appropriate IP address for each Pi.

### Step 3: Verify Connectivity

After setting up, I verified that all Pis could communicate with each other and with the internet by pinging them from one another:

```bash
ping 192.168.0.5
```

## Storage Headaches

One of the most significant challenges I faced was storage management within the cluster. With limited disk space on each Raspberry Pi, I needed to find a way to ensure that persistent data could be stored reliably and efficiently.

### Persistent Data Storage

To address this issue, I explored using external USB drives or network-attached storage (NAS) solutions. However, due to the limited bandwidth of the Ethernet connection between the Pis, I opted for an internal solution: configuring each Pi to use a portion of its SD card as persistent storage.

### Step 1: Prepare the SD Cards

I formatted the SD cards on one of the Pis and then used `rsync` to copy the necessary files across:

```bash
rsync -avz /path/to/kubernetes/ root@raspberrypi4-2:/mnt/persistent/
```

This command copied the Kubernetes configuration and other essential files to a directory named `/mnt/persistent`.

### Step 2: Mount Persistent Storage

On each Pi, I created a mount point for this persistent storage:

```bash
sudo mkdir -p /mnt/persistent
```

Then, edited the `fstab` file on each Pi to ensure that the SD card was mounted at boot time:

```bash
# /etc/fstab
/dev/mmcblk0p2  /mnt/persistent  ext4  defaults  0  2
```

### Step 3: Verify Mount

After rebooting, I verified that the persistent storage was correctly mounted:

```bash
df -hT | grep persistent
```

## Running Applications on the Cluster

With networking and storage issues addressed, it was time to deploy some applications. Given the limited resources of my Raspberry Pis, I decided to start with a simple application: a small web server running in Kubernetes.

### Step 1: Deploy a Web Server

I used `nginx` as an example application:

```yaml
# nginx-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25.4-alpine
        ports:
        - containerPort: 80

---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
    - protocol: TCP
      port: 80
      targetPort: 80
```

### Step 2: Apply the YAML File

Using `kubectl`, I applied this configuration to the cluster:

```bash
kubectl apply -f nginx-deployment.yaml
```

### Step 3: Verify Deployment

I used `kubectl` commands to check the status of the deployment and services:

```bash
kubectl get pods
kubectl describe service nginx-service
```

## Conclusion

Setting up a small Kubernetes cluster on three Raspberry Pi 4 boards was an enlightening experience, filled with both challenges and rewards. While networking and storage management presented initial hurdles, these were ultimately overcome through careful planning and resource allocation.

Running applications like `nginx` on such constrained hardware demonstrated the flexibility of Kubernetes in managing resources efficiently. However, it also highlighted the importance of considering performance implications when deploying more demanding workloads.

For those interested in exploring similar projects or looking to set up a small home cluster for educational purposes, this setup can serve as a valuable starting point. Whether you're building a personal lab environment, testing new technologies, or simply learning about container orchestration, there's plenty to discover and learn from this project.

Happy coding!
