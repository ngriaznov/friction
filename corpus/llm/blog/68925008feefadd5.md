## From the Command Line to the Flow: Why We Switched from Gerrit to GitHub Pull Requests

For years, our five-person engineering team had a deeply ingrained relationship with Gerrit. It was *the* code review tool – the cornerstone of our development process, built on a foundation of command-line interactions and meticulous checklists. We knew it inside and out, and frankly, we were comfortable. But recently, we made a significant shift: we switched from Gerrit to GitHub pull requests for our code reviews. It wasn’t a decision taken lightly, and there was a good deal of internal debate. Now, six months in, I wanted to share why we did it, what we missed about Gerrit, and the surprising benefits we've found.

Initially, the push towards GitHub pull requests came from a desire for greater integration with our existing workflows. We were using GitHub for everything – issue tracking, project management, documentation hosting – so the idea of streamlining code review through a single platform seemed logical. However, we quickly realized that simply migrating our processes wouldn't be enough. We needed to understand *why* Gerrit had become so ingrained in our habits and what we genuinely valued about it.

Let’s start with what we missed. Gerrit was, fundamentally, a tool built for rigor. It demanded absolute precision. Every change request (a “tree”) needed to be meticulously formatted – a specific file structure, precise commit messages, and detailed descriptions were mandatory. The automatic checks were incredibly thorough, catching everything from whitespace inconsistencies to minor logical errors. This wasn't just about finding bugs; it was about forcing us to think critically about every line of code before it even reached another person’s eyes. 

We missed the feeling of absolute certainty Gerrit provided.  The checklists – generated automatically and displayed prominently - were a constant reminder of our responsibilities. The detailed, automated checks felt like an extra layer of protection against human error. It fostered a culture of extreme care, which we found incredibly valuable in our mission-critical software development environment.  We also appreciated the dedicated command-line interface. While some felt it was cumbersome, for us, it provided a level of control and focus that a browser-based tool simply couldn’t replicate. The ability to carefully stage changes, review them repeatedly, and iterate directly within the command line felt more deliberate than clicking through a web interface.

But here's where things shifted. As we transitioned to GitHub pull requests, several key aspects became noticeably easier – and frankly, *better* in many ways. 

Firstly, the simplicity of the workflow was transformative. No more wrestling with complex commands or meticulously formatting tree files.  We could simply create a pull request, describe our changes in the issue comments, and have colleagues review them directly within the GitHub interface. This drastically reduced friction and accelerated the entire process. 

Secondly, the visual aspect significantly improved collaboration. The ability to see exactly what was changed – highlighted diffs, inline commenting, and easy navigation through the code – made discussions much clearer and more efficient.  We found it far easier to pinpoint specific issues and discuss them in context than sifting through lengthy text descriptions. 

Thirdly, GitHub’s built-in integration with other tools was a game changer. We leverage GitHub Actions for CI/CD, and the pull request workflow seamlessly integrates with our issue tracking system. This eliminated the need for manual data transfer and streamlined our entire development pipeline.

Fourthly, while Gerrit's automated checks were exhaustive, they sometimes felt overly strict.  GitHub’s checks are configurable and adaptable. We can tailor them to our specific needs – prioritizing critical security vulnerabilities over minor stylistic issues. 

Finally, GitHub’s community features have fostered a more open and collaborative environment. The ability to easily discuss changes with other developers, even those outside of our immediate team, has been incredibly valuable.

Of course, the transition wasn't without its challenges. We had to re-teach ourselves how to effectively utilize pull requests, and we spent some time configuring GitHub’s settings to match our desired workflow.  But ultimately, the benefits have outweighed the initial learning curve. 

Looking back, it wasn’t about replacing Gerrit; it was about embracing a more modern approach to code review that aligns with our current needs and leverages the power of integrated tools. We still value the rigor and discipline that Gerrit instilled in us, but we've found that GitHub pull requests offer a simpler, more collaborative, and ultimately more efficient way to deliver high-quality software.
