**Uncovering a Mysterious Memory Leak: A Node.js Investigation**

As a developer working on a large-scale Node.js application, I've grown accustomed to the usual suspects of performance issues: CPU spikes, network latency, and database query timeouts. However, one day our production monitoring tools alerted us to an unusual problem – a slow memory leak in one of our long-running worker processes.

**Symptoms**

At first, we noticed a gradual increase in memory usage over several days. Our monitoring system reported that the process was consuming more and more RAM, but it wasn't causing any immediate issues like crashes or errors. This made it difficult to pinpoint the problem. We checked for any obvious mistakes, such as unclosed database connections or file descriptors, but everything seemed in order.

**Initial Investigation**

Our first step was to use Node.js's built-in `process.memoryUsage()` function to gather more information about the memory usage of our worker process. This revealed that the memory leak was not due to a simple resource leak like an unclosed socket, but rather something more complex. We then turned to our logging system to see if there were any clues in the logs.

```javascript
console.log(process.memoryUsage());
```

The output showed that the process was using a large amount of heap memory (over 90%), but the RSS (Resident Set Size) was relatively small, indicating that most of the memory was being used by the V8 JavaScript engine. This suggested that our issue might be related to garbage collection or object retention.

**Profiling and Memory Profiling**

To get a better understanding of what was happening, we turned to Node.js's built-in profiling tools: `node --prof` and `node --heap-prof`. These tools generated files containing detailed information about CPU and memory usage. After analyzing the output, we discovered that our process was allocating and deallocating memory at an alarming rate.

```javascript
// Enable CPU profiling
node --prof -e "require('module').load('./path/to/module')"

// Generate heap profile
node --heap-prof -e "require('module').load('./path/to/module')"
```

The heap profiles revealed that we had a significant number of objects being retained in memory, preventing them from being garbage collected. This was the first real clue – something was holding onto these objects for an extended period.

**Event Listeners and Timers**

With this new information, our attention turned to event listeners and timers, which are known to cause memory leaks if not properly managed. We used `process.on('memoryPressure')` and `setInterval()` to monitor the process's memory usage over time. This led us to suspect that an event listener was responsible for the leak.

```javascript
// Monitor memory pressure
process.on('memoryPressure', () => {
  console.log(process.memoryUsage());
});

// Intervals for monitoring memory usage
setInterval(() => {
  // ...
}, 1000);
```

**The Root Cause**

After hours of debugging, we finally found the culprit – an event listener that had been attached to a module's `close` event. This listener was never removed and was causing objects to be retained in memory, preventing them from being garbage collected.

```javascript
// Event listener on close
module.exports.close = () => {
  process.on('close', () => {
    // Never removed this listener!
  });
};
```

**Conclusion**

Tracking down a slow memory leak in a Node.js worker process required patience and persistence. We had to employ various tools, from built-in profiling functions to external monitoring systems, to get to the root cause of the issue. The key takeaways from our investigation are:

* **Always remove event listeners**: Event listeners can cause memory leaks if not properly managed.
* **Monitor your application's memory usage**: Regularly check for memory leaks and optimize your code accordingly.
* **Use built-in profiling tools**: Node.js provides excellent built-in profiling tools that can help you identify performance issues.

By following these best practices, we were able to identify the root cause of our slow memory leak and prevent it from causing further problems. This experience has reinforced the importance of monitoring and optimizing our application's performance in production environments.
