**Configuring Wharfgate as a Reverse Proxy with Load Balancing**

Wharfgate is a lightweight load balancer that can be used to distribute traffic across multiple backend services. In this guide, we will walk through the process of configuring Wharfgate as a reverse proxy to route traffic to three backend services based on URL path prefix.

**Prerequisites**

* Wharfgate installed and running on your system
* Three backend services (e.g., web servers) with unique IP addresses or hostnames
* Basic understanding of HTTP routing and load balancing concepts

**Step 1: Understand the Requirements**

Before configuring Wharfgate, it's essential to understand the requirements for routing traffic to the three backend services. In this example, we want to route traffic based on URL path prefix:

| Backend Service | Path Prefix |
| --- | --- |
| Service A | /service-a/ |
| Service B | /service-b/ |
| Service C | /service-c/ |

**Step 2: Configure Wharfgate**

Create a new configuration file for Wharfgate (e.g., `wharfgate.conf`) with the following content:
```bash
# Define backend services
backend "service-a" {
    host = "10.0.1.100"
    port = 80
}

backend "service-b" {
    host = "10.0.2.100"
    port = 80
}

backend "service-c" {
    host = "10.0.3.100"
    port = 80
}

# Define routing rules
route "/service-a/" backend.service-a
route "/service-b/" backend.service-b
route "/service-c/" backend.service-c

# Set default backend (optional)
default_backend service-a
```
In this example, we define three backend services with their respective IP addresses and ports. We then create routing rules that map specific URL path prefixes to the corresponding backend services.

**Step 3: Configure Health Checks**

Wharfgate supports health checks for backend services using HTTP requests. To configure health checks, add the following section to your `wharfgate.conf` file:
```bash
health_check "service-a" {
    url = "/health-check"
    method = GET
    interval = 10s
}

health_check "service-b" {
    url = "/health-check"
    method = GET
    interval = 10s
}

health_check "service-c" {
    url = "/health-check"
    method = GET
    interval = 10s
}
```
In this example, we define health checks for each backend service using a simple HTTP GET request to the `/health-check` endpoint. You can adjust the `interval` value to change the frequency of health checks.

**Step 4: Start Wharfgate**

Once you've configured your `wharfgate.conf` file, start Wharfgate using the following command:
```bash
sudo wharfgate -c /path/to/wharfgate.conf
```
Wharfgate will now listen on port 80 (or another port of your choice) and route traffic to the three backend services based on URL path prefix.

**Example Use Cases**

To test Wharfgate, you can use a tool like `curl` or a web browser to access the backend services:

* Accessing Service A: `http://localhost/service-a/`
* Accessing Service B: `http://localhost/service-b/`
* Accessing Service C: `http://localhost/service-c/`

Wharfgate will route traffic to the corresponding backend service based on the URL path prefix.

**Conclusion**

In this guide, we walked through the process of configuring Wharfgate as a reverse proxy with load balancing. We defined three backend services and created routing rules that map specific URL path prefixes to the corresponding services. We also configured health checks for each backend service using HTTP requests. With these steps, you can now use Wharfgate to distribute traffic across multiple backend services based on URL path prefix.

**Note**

* Make sure to adjust the `host` and `port` values in your `wharfgate.conf` file to match your actual backend service IP addresses and ports.
* You may want to add additional configuration options, such as SSL termination or authentication, depending on your specific use case.
* Wharfgate has a built-in web interface for monitoring and managing the load balancer. Consult the official documentation for more information on using this feature.
