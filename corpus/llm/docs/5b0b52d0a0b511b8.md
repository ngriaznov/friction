**Ledgerline Caching Layer Overview**

The caching layer in Ledgerline is designed to optimize database queries by storing frequently accessed data in memory, reducing the load on the database and improving application performance. This overview explains how the caching layer works, covering query result caching, cache invalidation on writes, and the difference between per-request and process-level cache scopes.

**Query Result Caching**

The caching layer in Ledgerline stores query results in a memory-based cache, which is implemented using a Least Recently Used (LRU) eviction policy. When a user runs a database query through Ledgerline, the library checks the cache for an existing result with the same query parameters before executing the query against the database.

Here's how it works:

1.  **Cache Miss**: If there is no cached result for the given query parameters, Ledgerline executes the query against the database and stores the resulting data in the cache.
2.  **Cache Hit**: If a cached result exists for the given query parameters, Ledgerline returns the cached result instead of re-executing the query against the database.

**Cache Invalidation on Writes**

When a user modifies data through Ledgerline (e.g., creates, updates or deletes an object), the cache must be invalidated to ensure that subsequent reads return the latest data. Ledgerline uses a combination of strategies to invalidate the cache:

1.  **Explicit Cache Invalidation**: When a write operation is executed, Ledgerline explicitly invalidates any cached results associated with the affected query parameters.
2.  **Cache Tags**: Ledgerline assigns tags to each cached result based on the query parameters and database objects involved. During writes, these tags are used to identify which cache entries need to be invalidated.

**Per-Request vs Process-Level Cache Scopes**

The caching layer in Ledgerline supports two types of cache scopes: per-request and process-level. These scopes control how long cached results remain valid across multiple requests or the entire process lifecycle:

1.  **Per-Request Cache Scope**: By default, Ledgerline uses a per-request cache scope, where each request has its own isolated cache instance. This approach ensures that each request sees a fresh copy of the data and prevents stale data from being returned.
2.  **Process-Level Cache Scope**: For applications requiring longer-lived cache instances (e.g., across multiple requests), Ledgerline allows configuring a process-level cache scope, which shares a single cache instance among all requests within the same process.

Key differences between per-request and process-level cache scopes include:

*   **Cache Lifetime**: Per-request cache scopes have shorter cache lifetimes, as each request has its own isolated cache. Process-level cache scopes have longer-lived caches, sharing a single instance across multiple requests.
*   **Cache Sharing**: With per-request cache scopes, no data is shared between requests. In contrast, process-level cache scopes enable cache sharing among all requests within the same process.

By understanding how Ledgerline's caching layer works, you can optimize your application for better performance and ensure that your users receive up-to-date information.

**Example Code**

Here's an example of using Ledgerline with both per-request and process-level cache scopes:

```python
from ledgerline import sessionmaker

# Create a new database session with the default (per-request) cache scope
Session = sessionmaker()

session = Session()

# Use the session to execute a query and retrieve results from the cache
result = session.query(User).filter_by(id=1).first()

# To use a process-level cache scope, create a custom session class that configures
# the process-level cache instance
from ledgerline import configure_cache

class ProcessLevelSession(Session):
    def __init__(self):
        super().__init__()
        configure_cache(process_level=True)

process_session = ProcessLevelSession()
result = process_session.query(User).filter_by(id=1).first()

```

In this example, the `ProcessLevelSession` class creates a custom session that configures the process-level cache instance. This approach allows you to share a single cache instance across multiple requests within the same process.

By following the guidelines outlined in this overview and implementing Ledgerline's caching layer in your application, you can efficiently manage query results and improve performance without sacrificing data consistency or accuracy.
