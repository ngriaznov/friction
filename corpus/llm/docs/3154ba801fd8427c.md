# Caching Layer Overview in Ledgerline

The caching layer in Ledgerline is designed to improve the performance of queries by storing frequently accessed results in a cache store. This layer provides several benefits, including reduced database queries, faster response times, and improved overall application responsiveness.

## Query Result Caching

Ledgerline's query result caching allows developers to store the results of previously executed queries in a cache store. When a query is executed, Ledgerline checks if the result is already cached by querying the cache store. If a match is found, Ledgerline returns the cached result instead of re-executing the original query.

This approach provides several benefits:

*   **Reduced database queries**: By storing query results in a cache, developers can avoid re-executing queries that have already been answered.
*   **Improved performance**: Query caching reduces the time it takes to retrieve data from the database, leading to faster response times and improved overall application responsiveness.

## Cache Invalidation on Writes

When data is written to the database using Ledgerline's ORM features (e.g., `update()`, `insert()`, or `delete()` methods), the cache store needs to be updated accordingly. This ensures that cached results are not returned for queries that rely on outdated data.

Ledgerline implements a cache invalidation mechanism to handle this situation:

*   **Cache keys**: Ledgerline assigns unique cache keys based on the query parameters and any other relevant metadata.
*   **Write event listeners**: When a write operation is executed, Ledgerline triggers a series of event listeners that update the cache store. These listeners ensure that any cached results associated with the updated data are removed or invalidated.

By invalidating the cache on writes, developers can be confident that cached results will not be returned for queries that rely on outdated data.

## Per-Request vs Process-Level Cache Scopes

In Ledgerline, caching is configured using two distinct scopes: per-request and process-level. Understanding the differences between these scopes is crucial for effective caching in your application:

### Per-Request Cache Scope

*   **Scope**: Each incoming request has its own cache instance.
*   **Lifetime**: The cache instance is valid only for the duration of the current request.
*   **Sharing**: Multiple requests share the same cache instance.

The per-request cache scope provides a good balance between caching and freshness. Since each request has its own cache, you can minimize the impact on other requests if data becomes stale.

### Process-Level Cache Scope

*   **Scope**: The entire application process shares a single cache instance.
*   **Lifetime**: The cache instance is valid until the end of the process or until it is explicitly invalidated.
*   **Sharing**: All processes share the same cache instance.

The process-level cache scope provides higher performance benefits by reducing the overhead associated with caching multiple requests. However, this approach requires careful consideration to avoid stale data and ensure that caches are updated correctly in response to changes.

## Best Practices for Using Ledgerline's Caching Layer

To maximize the effectiveness of Ledgerline's caching layer:

*   **Use caching judiciously**: Only cache frequently accessed queries or results.
*   **Monitor cache performance**: Keep an eye on cache hit rates, miss rates, and overall performance.
*   **Implement cache invalidation strategies**: Regularly update caches to reflect changes in your application data.

By following these guidelines and understanding the intricacies of Ledgerline's caching layer, you can effectively leverage caching to improve the performance and responsiveness of your application.
