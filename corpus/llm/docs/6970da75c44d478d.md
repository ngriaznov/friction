### Overview of Caching Layer in Ledgerline: A Python ORM Library

Ledgerline is a robust Object-Relational Mapping (ORM) library designed to simplify database interactions for developers while maintaining high performance and scalability. One critical component within Ledgerline is its caching layer, which significantly enhances query performance by reducing the need to repeatedly fetch data from the underlying database.

This conceptual overview will cover how Ledgerline's caching layer operates, focusing on three primary aspects: 
1. **Query Result Caching**
2. **Cache Invalidation on Writes**
3. **Scope of Cache (Per-Request vs Process-Level)**

### Query Result Caching

#### What is Query Result Caching?
Query result caching stores the results of database queries in a fast-access memory structure, such as an in-memory hash table or cache server like Redis, instead of directly querying the database every time. This reduces the load on the database and speeds up application response times.

#### How It Works
When a query is executed, Ledgerline checks if the result has already been cached. If it exists, the data is served from the cache; otherwise, the ORM fetches the data from the database and caches it before returning it to the caller. Subsequent requests for the same query will directly serve the cached results.

#### Configuration Example
```python
from ledgerline.orm import setup, Session

setup('sqlite:///example.db', cache_enabled=True)

session = Session()
# Cache a result
result = session.query(User).first()

# Subsequent queries are served from the cache if available
```

### Cache Invalidation on Writes

#### What is Cache Invalidation?
Cache invalidation is the process of marking cached data as outdated or invalid when changes occur in the underlying database. This ensures that the next time a query for this data is executed, it will fetch the updated information.

#### How It Works in Ledgerline
Ledgerline supports various strategies for cache invalidation:
- **Manual Invalidations**: Explicitly invalidate caches after performing write operations.
- **Automatic Invalidations**: Automatically trigger cache invalidation when certain database operations occur (e.g., insert, update, delete).

For example, if a `User` record is updated in the database, Ledgerline can automatically invalidate any cached results related to that user. This prevents stale data from being served and ensures consistency between the ORM’s cache and the database.

#### Example of Automatic Invalidation
```python
# Manually invalidating a specific cache entry after an update operation
session.invalidate_cache(user_id=1)

# Or using automatic invalidation hooks in Ledgerline
@session.after_write('User')
def on_user_updated(instance):
    session.invalidate_cache(user_id=instance.id)
```

### Scope of Cache (Per-Request vs Process-Level)

#### Per-Request Cache
A per-request cache is scoped to the duration of a single HTTP request or function call. Data cached in this manner persists only for the lifetime of that specific operation and is cleared once the request completes.

##### Use Case
Useful when data doesn't change frequently and can be refreshed within the same request context, e.g., fetching user profiles on an API endpoint.
```python
from ledgerline.orm import Session

session = Session()
user_profile = session.query(UserProfile).filter_by(user_id=123).first()

# Cache is per-request scoped; it's cleared once the request ends
```

#### Process-Level Cache
A process-level cache persists for the entire duration of a Python process. Data remains cached until the process terminates or explicitly invalidated.

##### Use Case
Suitable when data needs to be shared across multiple requests within a single application instance, e.g., caching frequently accessed static information like configuration settings.
```python
from ledgerline.orm import setup, Session

setup('sqlite:///example.db', cache_enabled=True)

session = Session()
user_profile = session.query(UserProfile).filter_by(user_id=123).first()

# Cache is process-level scoped; it persists across multiple requests within the same application instance
```

### Conclusion

Ledgerline’s caching layer offers a powerful mechanism to enhance performance by reducing database load and speeding up query responses. By understanding how query result caching, cache invalidation on writes, and the scope of caches work, developers can leverage these features effectively to build efficient applications.

Implementing these caching strategies in Ledgerline ensures that your application remains performant while maintaining data integrity and consistency across both local and distributed environments.
